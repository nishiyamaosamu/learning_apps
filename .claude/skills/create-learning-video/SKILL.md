# 学習動画の制作（Remotion / 集中ブルー / ナレーション同期）

この仕事は「型にコンテンツを流し込む」作業ではない。**シナリオ（語り）が背骨で、
各ページはその語りを最もよく伝える絵**だ。1ページずつ、そのページのあるべき姿を
考えてレイアウトを設計する。見応えのある動画は、ページごとに最適な見せ方が選ばれていて、
隣り合うページが同じ形をしていない。

守るべき土台は固定されている:
- **色はトークンだけ**（`src/design/tokens.ts` の colors）。ハードコード禁止
- **動きは `useCurrentFrame`/`interpolate` 系だけ**（`src/parts/animate.ts` のフック）。CSS animation/transition は禁止（レンダリングが壊れる）
- 本編ページは下部に**ナレーション同期字幕**を常設（`SlideShell` に narration を渡せば保証される）
- 文字は大きく・要素は少なく（動画は視聴距離が遠い）
- **尺は1本5分以内が目標。5:59（359秒）まではセーフ、6分以上はアウト**。原稿の文字数で管理する

作業ディレクトリ: `content_works/atelier/video`

## ワークフロー

### 1. 手本と道具を知る

最初に必ず読む:
- `src/videos/demo/narration-demo.tsx` — ナレーション＋字幕同期の最小手本。**必読**
- `src/videos/demo/demo.tsx` — 全パターンの記述例 + カスタムシーンの手本（StatScene）
- `references/custom-scenes.md`（このスキル内）— カスタムシーンのレシピ集。**必読**
- `references/narration.md`（このスキル内）— 音声パイプラインの詳細（コマンド・命名・部分再生成・トラブルシュート）。工程4の前に読む

### 2. シナリオライティング（語りが背骨）

実装より先に、レッスン教材から「語り」を書き切る。動画の質と尺はここで決まる。

1. **構成**: title → 導入（なぜ学ぶか・身近な例）→ 本編 → クイズ導入の幕間 → クイズ2〜3問 → まとめ。
   全体で12〜16ページ
   - title はオープニングジングル（6.8秒）が自動で鳴る。**title にナレーションは付けない**
   - クイズの前に幕間ページを1枚置いて一拍おく。画面は `parts/SectionTitle.tsx` で
     短い題の文字アニメ（例: 「クイズで確認」）、読み上げは
     「ここまで学んだことをクイズ形式で確認していきましょう！」のように長くてよい
   - まとめは本編の要点を短く復唱し、最後のセグメントは必ず
     「今回のレッスンはここまでです。お疲れさまでした！」で締める
   - **まとめの頭にはワイプ**: まとめページの spec に `transitionIn: "wipe"` を付ける
     （太めの角丸ラインが画面を埋めてから抜ける約1.5秒の切り替え）。**まとめ前専用・1動画1回**。
     本編内で話題が大きく変わるページには軽量版 `transitionIn: "wipe-light"`
     （斜めの平行四辺形3枚が流れ抜けるだけ・約1.0秒）を使ってよい（1〜2回まで）。
     クイズ導入の幕間にはどちらも付けない（SectionTitle が主役）
2. **キーワードは説明してから先へ進む**: 画面に出す用語（例: ミッション・ビジョン・バリュー、
   グリーンIT、カーボンフットプリント）は、名前を読み上げるだけで流さず、それぞれ一言の定義か
   身近な例を必ず語る。用語の羅列は記憶に残らない。語る時間が足りないなら項目ごと削って
   残りを深く — **網羅より記憶**
3. **各ページのナレーション原稿**を書く。話し言葉（〜です・〜ます）で、耳で聞いて自然な文。
   Markdown記号や「・」の読みは書かない。字幕にもそのまま出る文なので、読んでも聞いても通じること
4. **1文 = 1セグメント = 字幕1枚**。1文は25〜45字、1ページ2〜6文。長い文は必ず区切る
   （字幕1枚に収まらない文は、聞いていても長い）
5. **原稿の総文字数で尺を管理する**: 読み上げは約4.5〜5字/秒。目標は **1,200〜1,400字（4〜5分）**。
   1,500字を超えたら音声を作る前に削る — ページごと削るのが効く。作った後に削ると再生成コストがかかる。
   逆に3分を大きく下回るなら、キーワードの説明が浅くないか（工程2の観点で）見直す
6. 成果物を2つ書く:
   - `narration/<id>.md` — シナリオ（ページ表: メッセージ・レイアウト案・原稿）。後から見返す用
   - `narration/<id>.jobs.json` — セグメント原稿（`s{ページ2桁}-{通し}.mp3` → 文）。TTSの入力

### 3. ページ設計（原稿を絵にする）

シナリオの各ページについて、次の3つを短くメモに書き出す:

1. **メッセージ**: このページで伝えることを一文で。二文になるならページを分ける
2. **あるべき姿**: 視聴者が最初の0.5秒で何を見るべきか（大きな数字？ 対比？ 循環？ 積み上がる構造？ 放射状の関係？ 1つの単語？）
3. **形**: あるべき姿をそのままレイアウトにする（**custom が基本**）

**画面テキストは原稿の要点だけにする。** 字幕がナレーション全文を語るので、
画面に原稿と同じ文を書くと二重になる。画面側はキーワード・図・関係だけを引き受ける。

**custom が基本、パターンは近道。** 循環図・ハブ&スポーク・ピラミッド・数字ドン・カード比較など、
内容が求める形をゼロから設計するのがこの環境の標準的な使い方だ。既製パターン(下表)は
頻出形の実装済みショートカットで、内容がその形に完全に一致するときだけ使ってよい。
「パターンに収まるよう内容を曲げる」のは本末転倒。

| パターン | 形 |
|---|---|
| `bullets` | 並列な列挙（3点まで） |
| `vs` | 2概念の対比 |
| `flow` | 順序ステップ・サイクル |
| `matrix` | 2軸4象限 |
| `layers` | 階層・積み上げ |
| `graph` | 図と式のペア（座標系は references/patterns.md） |
| `term` | 用語ドン（1動画1〜2回まで） |
| `binary` | 桁ごとの計算ステップ |
| `quiz` | 理解確認（問→答の間に考える間を置く。references/narration.md） |

**リズムの設計**（絵コンテ全体を見て確認する）:
- 同じ形（同じ視覚文法）を2ページ連続させない。循環図の直後に放射図など「線で繋ぐ図」を重ねない
- 密なページの次は疎なページ（数字ドン・用語ドンなど）で息継ぎを作る
- ナレーションが長いページ（4文以上）は、要素を時間差で順に出して「語りに画面がついてくる」感を作る

**イラストを入れるか考える**（ページ設計の一部として毎回検討する）:

手描き風イラスト（スタイル見本: `docs/SAMPLE_STYLE.png`）には、抽象的で難しい話を
易しく感じさせ、イメージを膨らませる効果がある。各ページの「あるべき姿」を考えるとき、
「これは絵で見せた方が入ってくるか？」を必ず一度問う。

- **向いている**: 情景や人物が絡む話（オフィスの場面・攻撃者と守る人）、抽象概念のメタファー
  （信頼＝握手、責任＝天秤、資源＝集まってくる矢印）、導入や締めで感情に訴えたいページ
- **不要**: 構造図・計算・チャートなど、動画側の SVG／レイアウトで描ける形（それは custom で組む）
- **分量**: 1動画 2〜4枚まで。多用すると1枚の効果が薄れる

イラストを使うときの手順:
1. **まず既存部品を探す**: `content_works/<app>/video/assets_common/`（`icon/`・`images/`）と
   動画プロジェクトの `public/images/`。同じ概念には必ず同じ絵を使い回す（記憶のフック）
2. 無ければ **create-video-illust スキルを Skill ツールで呼んで生成する**（1枚数分・必ず直列)。
   `<app>` はレッスンが属するアプリ（例: ipa_ip）。透過処理まで済ませる。
   codex はフォアグラウンドで実行し、**「バックグラウンドの完了待ち」を理由に作業を止めない**
   （待ち時間には他ページの実装・lint・stills を進める）。イラストが揃わないときは
   該当ページをイラストなし設計に切り替えて完成を優先する
3. 生成物は `assets_common/draft/` に落ちる。動画で使う分だけ `public/images/` に **cp する**
   （draft の原本は動かさない — 共有部品への本採用はユーザー承認が要る）
4. シーンでは `<Img src={staticFile("images/xxx.png")} style={{ mixBlendMode: "multiply", ... }} />`
   で白地を背景に溶かす（references/custom-scenes.md の左右分割レシピ参照）

### 4. 音声生成（実装の前に済ませる）

詳細・部分再生成・トラブルは `references/narration.md`。基本の流れ:

```bash
cd content_works/scripts
mise exec -- uv run tts.py --jobs ../atelier/video/narration/<id>.jobs.json \
    --out-dir ../atelier/video/public/audio/<id>

cd ../atelier/video
node scripts/audio-durations.mjs <id>   # 実測秒数 → src/videos/<id>.audio.json
```

- audio-durations が合計秒数を報告する。**警告（330秒超）が出たらここで原稿を削る** —
  実装後に削るより圧倒的に安い
- 実装は `<id>.audio.json` を import するので、音声を先に作らないと型チェックが通らない（意図的な順序強制）

### 5. 実装

`src/videos/<id>.tsx` を作り、`src/videos/index.ts` の `videos` 配列に追加する。
ナレーションの組み立ては `narration-demo.tsx` の形をなぞる:

- `narrationLoader(durations, "audio/<id>")` で `N()` を作り、セグメント配列を組む
- **既製パターン**: spec に `narration: [...]` を書くだけ（telop は書かない）
- **カスタム**: セグメント配列を定数にして **spec と SlideShell の両方に同じものを渡す**
  （spec 側が音声と尺、SlideShell 側が字幕。片方だけだと音か字幕が欠ける）
- custom の `durationSec` はアニメーションの最低尺。ナレーションの方が長ければ自動で伸びる
- 音声はレンダラーが自動で鳴らす。シーン内に `<Audio>` を手動で置かない（二重再生）
- title はジングルが自動で鳴る（ナレーションなし）。ダーク幕間は中央の大きな文字を読み上げ文と一致させる
- クイズは正解セグメントに `gapBeforeSec: 1.5〜2` を付けて考える間を作り、
  `revealAtSec` をその正解セグメントの開始秒（前セグメント合計 + gap）に合わせる
- 章の切り替えは spec の `transitionIn`: まとめ前は `"wipe"`（専用）、本編内の場面転換は `"wipe-light"`
  （自前でワイプを実装しない — parts/transition.tsx が正）

**画面テキストの分量上限**（超えそうなら削るのではなくページを分ける）:

| 場所 | 上限 |
|---|---|
| title | 2行・各行13文字 / keywords 3個・各7文字 |
| bullets | 3項目 / text 14文字 / sub 20文字 |
| vs.rows | 各列3行 / 左右で観点(k)を揃える |
| flow.steps | 3〜5個 / name 4文字 |
| quiz | question 20文字 / 選択肢 text 18文字 / correct は1つ |
| 字幕セグメント（全ページ） | 1文25〜45字・話し言葉（〜です・〜ます） |

**カスタムシーンの決まり**（詳細とレシピは references/custom-scenes.md）:
- 本編ページは `<SlideShell narration={...}>` で包む（字幕と余白の保証）。幕間だけ例外
- 文字サイズは `videoType` か `モック値 × SCALE`（最小 9×SCALE=36px）
- 出現アニメーションは `useAppear` / `usePop` / `useFlyIn` / `useProgress` を時間差で
- 線・矢印の図解は `parts/draw.tsx` の `DrawPath` / `ArrowMarker`（SVGの手計算を減らす）
- アイコンは `<Ms name="..." />`。名前は `references/icons.md` のリストから
- クイズ的なページを自前設計する場合も、正誤は**色 + ○記号の二重符号化**（colors.correct 系）を守る

### 6. 機械検証

```bash
cd content_works/atelier/video
node scripts/lint-videos.mjs   # デザイン違反の検出（ハードコード色・CSSアニメ等）
npx tsc                        # 型チェック
```

両方通るまで先に進まない。

### 7. 全ページを静止画で目視確認(省略禁止)

```bash
node scripts/stills.mjs <id>
```

`stills/<id>/` の**全ファイルを Read で開いて見る**。チェック観点:

- はみ出し・折り返し・重なりがないか / アイコンが文字列のまま出ていないか
- 字幕帯に文が出ているか（telop の書き忘れ・narration の渡し忘れはここで見える。
  なお stills はシーン末尾なので、写るのは各ページ最後のセグメントの字幕）
- **単調でないか**: 並べて見て、同じ視覚文法が連続していたら設計(工程3)に戻る
- **0.5秒で伝わるか**: そのページのメッセージが一目で入ってくるか
- スカスカ/ぎゅうぎゅうのページはないか

とくに図解系のカスタム（グリッド・SVG座標もの）は、要素のあふれ・重なりが lint/tsc では
検出できず**静止画でしか見つからない**。ここで1回は直しが入る前提で時間を使う。
静止画で確認せずに完成報告するのは、このワークフロー最大の禁止事項。

### 8. レンダリングして納品

```bash
npx remotion render <id> draft/<id>.mp4
ffprobe -v error -show_entries format=duration -of csv=p=0 draft/<id>.mp4   # 359秒以下を確認
ffprobe -v error -show_entries stream=codec_type -of csv draft/<id>.mp4     # audio ストリームがあること
```

成果物は必ず `draft/` へ。**359秒を超えていたら納品しない** — 原稿を削って該当音声だけ
再生成し（references/narration.md）、作り直す。
完了報告にはページ構成（各ページの形とメッセージ）・合計秒数・原稿の総文字数を含める。
