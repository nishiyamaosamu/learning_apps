# 動画用イラスト部品の生成

学習動画（`content_works/<app>/video/`）のシーンに合成する部品イラストを作る。
動画レイアウト側で矢印・枠・テキストを重ねる前提なので、部品は「1部品＝1オブジェクト」のシンプルな絵にする。
どの部品を作るかはプロジェクトごとの計画書（例: `content_works/ipa_sg/ILLUST_PLAN.md`）が持つ。本スキルは作り方だけを扱う。

## スタイル規定（全プロジェクト共通）

- スタイル見本は `docs/SAMPLE_STYLE.png`（リポジトリルート基準）。**生成プロンプトで必ずこのファイルを参照させる**
- 細めで少し揺らぎのある黒い手描き線＋色鉛筆風の塗り。柔らかく明るい日本語インフォグラフィック調
- 塗りは白に飛ばない程度にはっきり発色させる（淡すぎると白背景と区別がつかず、透過処理でも抜けやすい）
- 説明テキストは入れない（単語ラベル程度はOK）
- 色に意味を持たせる場合の対応表はプロジェクトの計画書側で定義し、プロンプトに含める（例: SGは青＝守る側／赤＝攻撃者／緑＝OK／黄＝注意）
- サイズ: 単体アイコンは 1:1（1024×1024）、シーン絵は 16:9 または 4:3

## 配置と命名（全プロジェクト共通）

```
content_works/<app>/video/
├── assets_common/     # 全レッスンから使い回す共通部品
│   ├── draft/         # 生成直後の確認待ち画像（本採用前は必ずここ）
│   ├── icon/          # `icon-` 接頭辞の単体アイコンのみ（1:1）
│   └── images/        # それ以外の部品すべて（人物・シーン・脅威/技術/管理などの部品）
└── assets_l{n}/       # レッスン固有の一点物（heroなど）
```

- ファイル名: `{カテゴリ}-{名前}.png`（例 `icon-shield.png`, `person-attacker.png`, `scene-office.png`）
- 本採用フォルダの振り分け: `icon-` 接頭辞は `icon/`、それ以外の接頭辞（`person-` `scene-` `malware-` `attack-` `tech-` など）はすべて `images/`
- 状態違いは接尾辞: `-ok` / `-ng` / `-open` / `-broken`（例 `icon-lock-open.png`）
- 同じ概念には必ず同じ部品を使い回す（絵の一貫性＝学習者の記憶のフック）
- **生成した画像は本来のフォルダ（`icon/` `images/` `assets_l{n}/`）に直接置かない。** 必ず一旦 `assets_common/draft/` に格納し、ユーザーから「本採用フォルダに移して」等の指示を受けてから対象フォルダへ`mv`する

## ワークフローA: 単体部品（アイコン・シーン絵・小物）

### 1. codexで生成

リポジトリルートから以下の雛形で実行する。1件あたり数分かかる。
**複数作るときも codex は直列（1件ずつ）で投げること。** 並列に投げると成功率が下がり、
レイアウト途中段階の未完成画像が出力される事故が増える。前の1件が完了してから次を投げる。

```
codex exec --sandbox workspace-write "
学習動画のシーンに合成する単体イラスト部品です。
スタイルの見本: docs/SAMPLE_STYLE.png を必ず参照し、この画風（細めで少し揺らぎのある黒い手描き線、白背景、淡い青・緑・黄・ピンクの色鉛筆風の塗り、柔らかく明るい日本語インフォグラフィック）に合わせてください。見本はスタイルシートなので、構成やラベルは再現しないでください。
背景は白または透過で、オブジェクト1つだけを描き、説明テキストは入れないでください（単語ラベルはOK）。
色の意味: <計画書の色対応をここに書く>
塗りは白に飛ばない程度にはっきり発色させること。
サイズは<1024x1024 / 16:9など>。
保存先: content_works/<app>/video/assets_common/draft/<部品名>.png
内容:
<部品の内容を日本語で簡潔に。計画書の「内容」列をベースに具体化する>
"
```

雛形の意図（省略・改変するときの判断基準）:
- 「見本はスタイルシートなので構成を再現しない」— これがないと見本のグリッド構成やラベルごと模倣される
- 「はっきり発色」— 色鉛筆調は放っておくと淡くなりすぎ、ほぼ白の塗りになる事故が多い

### 2. 結果を目視確認

生成されたPNGをReadで開き、受け入れ基準を確認する:

- 依頼した概念が一目で伝わる（焦点が1つ）
- 画風が見本と揃っている（線の太さ・塗りの質感）
- 塗りの発色が十分（白に飛んでいない）
- 余計なテキスト・複数オブジェクト・枠がない
- 色の意味対応が正しい

基準を外れたら、外れた点を具体的に指摘する行をプロンプトに足して再生成する
（例: 「前回の生成は塗りが淡すぎてほぼ白に見えたため、青の発色を一段濃くすること」）。
見本内の具体箇所を参照させると効く（例: 「見本①の『ヒト』の円の青と同じくらい」）。

### 3. 背景を透過にする

画像生成は実質白背景でしか出せないため、生成後に必ず透過処理を通す（draftフォルダ内で処理する）:

```bash
cd content_works/scripts
mise exec -- uv run remove_bg.py ../<app>/video/assets_common/draft/*.png
```

- 外周フラッドフィル方式なので、オブジェクト内部の白（書類の紙面・ハイライト等）は保持される
- 透過済みの画像は自動スキップされる。ディレクトリ丸ごとワイルドカードで実行してよい
- **codex は完了間際に画像を再保存することがある**（確認・リサイズのため）。codex の実行が
  完全に終わってから透過処理すること。処理後にアルファチャンネルが残っているか確認し、
  RGB版で上書きされていたら再度 remove_bg を通す
- 制約: 鍵穴のような「囲まれた穴」は白のまま残る。濃い背景に載せる部品で気になる場合のみ個別対応

透過の確認が必要なら、市松模様に合成した確認画像を作ってReadで見る（アルファ値だけでは縁の抜け残りが分からない）。

### 4. 本採用フォルダへの移動

draft内の画像はここでは仮置きのまま。ユーザーが目視確認して「これでOK」「本番フォルダに移して」等の指示を出したら、
該当ファイルを振り分け規則に従ってフォルダへ`mv`する（`icon-`→`icon/`、それ以外→`images/`。例 `mv assets_common/draft/icon-shield.png assets_common/icon/`、`mv assets_common/draft/scene-office.png assets_common/images/`）。
指示が来るまでdraftに置いたままにし、先回りして本来のフォルダへ移動しない。

## ワークフローB: 人物キャラ（ポーズ違い）

人物は1ポーズずつ生成するとキャラの同一性が崩れる。**1キャラ＝1枚のポーズシート**として
複数ポーズを1回で生成し、スクリプトで分解する。

1. codexでポーズシートを生成（ワークフローAの雛形を流用。保存先はdraft、内容に「同一人物の
   ポーズ違いをグリッド状に並べる。各ポーズの間は空ける」と各ポーズの説明を書く）
2. 分解＋背景透過してdraftに展開:

```bash
cd content_works/scripts
mise exec -- uv run split_pose_sheet.py \
    --src <ポーズシート.png> \
    --names "名前1,名前2;名前3,名前4" \
    --out-dir ../<app>/video/assets_common/draft
```

`--names` は「,」区切りが列、「;」区切りが行（読み順）。基本ポーズは接尾辞なし、
バリエーションは `-point` `-worry` など意味の分かる接尾辞にする（計画書の表に合わせる）。
3. 目視確認後、ユーザーの指示を受けてから該当ファイルを`assets_common/images/`へ`mv`する（ワークフローAの手順4と同様）。

## 完了時

- 本採用フォルダへの移動はユーザーの確認指示があったものだけ。draftに残っているものは未確定として扱う
- 移動した部品は計画書の表の部品名どおりのファイル名で配置されているか確認する
- 計画書に進捗マーク（✅ 作成済み など）を付ける運用ならそれも更新する
