# 情報セキュリティマネジメント試験（SG） 暗記カード計画（よく出る単語だけ!）

全体目標：全8グループ **29デッキ・332枚** の暗記カードで構築する。暗記タブの看板どおり
「**よく出る単語だけ!**」— シラバスVer.4.1の用語例（延べ約800語）を網羅するのではなく、
公開問題（令和5〜8年度）の出題実績と反復論点を根拠に厳選する。
[LESSON_PLAN.md](LESSON_PLAN.md) の【地図＋代表】レッスンが動画で扱わない用語の
「受け皿」を兼ねるが、委譲分も網羅ではなく**厳選し、落とした用語は本書末尾で明示**する
（除外・吸収リストにより、シラバスとの突き合わせを監査可能にする）。

- 参考資料：[syllabus_sg_ver4_1.md](syllabus_sg_ver4_1.md)（出題範囲・用語例の表記の正）／
  [sg_koukai_mondai.md](sg_koukai_mondai.md)（公開問題 R5〜R8。★★★タグの根拠）／
  [LESSON_PLAN.md](LESSON_PLAN.md)（章構成・委譲元・用語の主担当）

## 分量・粒度基準（1デッキあたり・厳守）

- 1デッキ ＝ **8〜15枚**（目標10〜14）。スキマ時間で1周できる粒度にする。
- カードは `{front, back}` のプレーン文字列のみ（engineの `AnkiCard` 仕様）。装飾・改行・画像は使えない。
- back は **60〜100字・1〜2文**。第1文＝定義、第2文（任意）＝試験での識別ポイント。
- デッキファイルは `apps/ipa_sg/contents/anki/{id}.json`、一覧は `base.json` の
  `anki`（グループ → デッキ）に登録する。`cardCount` は実カード数と機械的に一致させること。

## 設計方針

- **絞り込み3原則**：
  1. **公開問題実績を最優先**：R5〜R8の科目Aで出題された用語・論点（★★★）は必ず収録する。
  2. **委譲分は網羅でなく厳選**：【地図＋代表】レッスンからの委譲用語は重要度で選別し、
     落とした用語は「収録しない用語」節に除外理由つきで明示する。
  3. **計算・シナリオはカード化しない**：計算が本体の論点（稼働率・サブネット・損益分岐点など）は
     定義カードのみ残し、計算ドリルは問題集タブへ。科目B（シナリオ判断型）はカード化しない
     （L78〜L96の動画と問題集が担当）。
- **優先度タグ**（全カードに付与し、絞り込みを監査可能にする）：

  | タグ | 基準 | 枚数目安 |
  |---|---|---|
  | ★★★ | R5〜R8公開問題（科目A）で出題実績のある用語・論点 | 約50 |
  | ★★ | 反復出題が予想される定番（重点分野の中核用語・対概念ペア・R7以降増加傾向の新技術/DX系） | 約225 |
  | ★ | 【地図＋代表】委譲の受け皿などから厳選した最低限カバー | 約55 |

- **除外と吸収の2段構え**：収録しない用語は「**除外**」（完全に落とす）と「**吸収**」
  （独立カードにせず、関連カードの back で一言触れる。例：トラストアンカー→ルート証明書の back）に分ける。
  どちらも本書末尾のリストで管理する。吸収指定はカード執筆時に back へ必ず反映すること。
- **束ねルール**：セットで初めて意味を持つ語は1枚に束ねる（RTO/RPO/RLO、SaaS/PaaS/IaaS、
  リスクの4損失、偽・誤情報の3類型など）。逆に対概念は別カードにし、back で相互の違いに触れる
  （FRR/FAR、フォールスネガティブ/ポジティブ、差分/増分バックアップなど）。
- **1用語1枚・重複収録禁止**：章をまたぐ用語（VPN・パケットフィルタリングなど）は1デッキだけに置き、
  back で関連に触れる。
- **呼称は LESSON_PLAN「用語の主担当（定義の正）」に統一**：匿名加工の手法名＝L50、
  バックアップ方式＝L59、RASIS＝L56、RTO/RPO/RLO＝L69 などの呼称に合わせる。
- **並び順**：デッキ内のカードは対応レッスンの登場順に近づける（動画→カードの復習動線）。

## カード執筆ガイドライン

- **front ＝ 用語のみ**（問いかけ文にしない）。表記はシラバス用語例に準拠する。
- **back に答えを書かない**：アプリは back（説明）を先に表示し、フリップで front（用語）を出す。
  back に front の用語・略称の正式名称展開が入っていると答えがバレるため禁止。
  用語側にだけ見せたい補足は `frontInfo`（フリップ後に用語の下へ小さく表示）に置く。
  - 英略語の正式名称展開（「〜の略」）は back でなく frontInfo に書く（英語展開は頻出のもののみ）。
  - 対概念・列挙を束ねたカードは back で個々の名前を出さず、名前と説明の対応関係を frontInfo に書く
    （例：`"frontInfo": "送信＝SMTP、受信＝POP3・IMAP"`）。
  - ただし front 自体が構成要素の列挙（ポリシー3階層など）や、用語が定義そのものを含む合成語
    （サーバ証明書・損益分岐点など）は、back に要素名が出るのを許容する。
- 法律カードは「何を規制・処罰する法か＋識別ポイント」、機関カードは「どこの・何をする組織/制度か」の型で書く。
- 例カード：

```json
{ "front": "SIEM", "back": "サーバやネットワーク機器のログを一元的に収集・相関分析し、攻撃の兆候をリアルタイムに検知する仕組み。", "frontInfo": "Security Information and Event Managementの略" }
{ "front": "アンチパスバック", "back": "入室記録のない人の退室や再入室を拒否する入退室管理方式。共連れによる不正入室への対策になる。" }
{ "front": "残留リスク", "back": "リスク対応を実施した後になお残るリスク。ゼロにはできないため、受容基準内かどうかをリスク所有者が判断して受け入れる。" }
```

## base.json 追加構造（実装時のスケッチ）

グループ title は LESSON_PLAN の章名に対応させる。デッキIDはグループ番号-連番で、
ファイルは `apps/ipa_sg/contents/anki/{id}.json`（例 `1-1.json`）。
`cardCount` は実カード数と一致させること（subtitle フィールドは存在しない）。

```json
"anki": [
  {
    "id": "g1",
    "title": "脅威と攻撃",
    "anki": [
      { "id": "1-1", "title": "情報セキュリティの基本特性", "cardCount": 10 },
      { "id": "1-2", "title": "マルウェアの種類", "cardCount": 11 },
      { "id": "1-3", "title": "人・パスワードを狙う攻撃", "cardCount": 10 },
      { "id": "1-4", "title": "Web・通信への攻撃", "cardCount": 14 },
      { "id": "1-5", "title": "攻撃の流れと新しい脅威", "cardCount": 11 }
    ]
  },
  { "id": "g2", "title": "暗号と認証", "anki": [ "…以下同形式…" ] }
]
```

グループ一覧：g1 脅威と攻撃（56）／g2 暗号と認証（29）／g3 情報セキュリティ管理（55）／
