# レッスン1ページの分量ガイド（iPhone 17基準）

レッスンのコンテンツページが「1ページに収まるか」を、**作る前の設計指針**と
**作った後の機械チェック**の両面で扱うためのドキュメント。

## 大前提：はみ出さない、代わりに「縮む」

レッスンカード（`_LessonCard`）は本文を `FittedBox(fit: BoxFit.scaleDown)` で
描画する（[lesson.dart](../engine/lib/src/screens/tabs/lesson.dart) の `_buildContent`）。
そのため本文が領域を超えても画面からはみ出すことはなく、**文字・画像をまとめて
一律に縮小して**1ページへ押し込む。

→ 判定すべきは「はみ出したか」ではなく **「縮小されたか（= 分量オーバー）」**。
本文の原寸の高さが、カードが原寸表示で使える高さ（予算）を超えると縮小される。

## 1ページの予算（iPhone 17 / 論理 402 × 874pt）

レイアウトから逆算した、本文が**原寸（縮小なし）で使える領域**：

| 項目 | 値 |
|------|----|
| 本文の高さ予算 | **約 613pt** |
| カード幅 | 370pt |
| 本文の横幅（左右padding 20差引後） | **330pt** |

寸法の実体は `LessonCardMetrics`
（[lesson_contents.dart](../engine/lib/src/screens/tabs/widgets/lesson_contents.dart)）。
ヘッダー56 / フッター72 / セーフエリア上59・下34 / ページ控除40 などから算出される。

### 各要素の概算コスト（613ptの予算をどれだけ食うか）

| 要素 | 概算高さ |
|------|---------|
| ページタイトル（+下の余白） | 約 33pt |
| 本文テキスト 1行 | 約 24pt |
| ブロック間の余白 | 16pt / 個 |
| 画像（16:4.5 横長） | 約 93pt |
| `:::box`（枠） | 枠で +32pt ＋ 中身 |
| `:::iconlist` 1項目 | 約 58pt |

## 設計時の目安（作る前）

あくまで**当たり付け**。マークアップ次第で高さは大きく変わるので、確定は後述の
チェックツールで行う。

- **画像なしのページ**：本文は合計 **全角250字以内**（usage 85%前後）を目安。
- **画像1枚を入れるページ**：画像が約15%を食うので、本文は **全角150字以内**。
- **ブロックは3〜4個まで**。増やすほどブロック間余白（16pt）で損をする。
- **`:::iconlist` / `:::box` / 見出しは高さを大きく食う**。使うなら本文テキストを削る。
  `:::iconlist` 4項目だけで約230pt（予算の約4割）を消費する。
- 字数が同じでも、`:::box` や見出しを使うと usage が30ポイント以上跳ね上がることがある。
  迷ったら必ず実測する。

## 作った後のチェック

### A. 実機デバッグバッジ（その場で正確に確認）

デバッグビルドでは、レッスンカードの**右上**に usage%（本文の原寸高さ ÷ 613pt）が
表示される（`_FitBadge`）。**実機の本物のフォントで測るため、これが最も正確**。

- **緑** = 90%未満 / **橙** = 90〜100% / **赤** = 100%超（実機で縮小）
- リリースビルドでは表示されない

### B. 自動テスト（CIゲート）

`apps/ipa_ip/test/page_fit_test.dart` が、実際の描画と同じ
`buildLessonCardColumn` / `MarkdownText` で各ページを組み、原寸の高さを計測して
**usage（高さ ÷ 613pt）** を一覧表示する。縮小されるページがあるとテストが失敗する。

```
cd apps/ipa_ip && mise exec -- flutter test test/page_fit_test.dart
```

出力例：

```
  p 8   538pt   88%  OK       | 文字54+文字228
  p 9   629pt  103%  NG（縮小）   | 文字35+文字177+画像
```

- **OK**：usage 90%未満。余裕あり。
- **ギリギリ**：90〜105%。実機で縮む可能性があるため、削るのが安全。
- **NG**：105%超。要修正。

機種を変えて確認したい場合は、テスト冒頭の `_screen` / `_viewPadding` を差し替える。

### 精度について（なぜ NG 判定が 105% なのか）

Material 3 のタイポグラフィは行高を明示指定しているため、テスト用フォントでも
**行高は実機と一致**する。一方、字送り幅は日本語（全角）ではほぼ一致するが、
ラテン文字では**やや過大評価**するため、テストの usage は実機より数%高く出る
（実測: レッスン1 p9 は実機バッジで 99% だが、テストでは 103%）。この過大評価を
吸収するため、テストの NG 判定は 105% 超としている。**正確な値が必要なときは実機の
デバッグバッジを見ること。**
